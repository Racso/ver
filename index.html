Datetime Versioning

Ship dates,
not numbers.

Version numbers that tell you when something was built — automatically, monotonically, and without anyone having to think about it.

The problem

SemVer exists to communicate backwards compatibility. That's useful for libraries — but most software isn't a library. A single-player game doesn't have a "backwards-compatible API." A multiplayer game might have multiple internal systems that need their own compatibility tracking, but the game itself doesn't fit the SemVer model.

In practice, when teams are building, iterating, testing, and shipping, the questions they actually ask about versions are:

  • "Is this screenshot from the latest build or an old one?"
  • "Does this build include the fix I pushed yesterday around 3pm?"
  • "Which of these two versions came first?"

These are questions about time, not compatibility. Ver answers them.

Design goals

1. Don't get in the way. No one should have to decide what the next version number is. No debates about whether this is a minor bump or a patch. The build timestamp is the version.
2. Fully automatable. Generate the version from the current time. No state files, no incrementing counters, no coordination between CI jobs. Guaranteed monotonically increasing.
3. Meaningful information. A version number should tell the team something useful at a glance. "When was this built?" is more useful than "what arbitrary number did we pick?"

The family

SunVer
The version is the date. No encoding, no math.
X = year   Y = MMdd   Z = HHmm
range unlimited resolution 1 min legibility perfect MSI no
LightVer
SunVer's constrained sibling. Fits within MSI limits.
X = year − 2000   Y = month   Z = dd×1000 + HH×10 + m
range 255 years resolution 10 min legibility high MSI yes
MoonVer
Minute precision under MSI constraints. One ÷20 to decode.
X = offset×20 + month   Y = day   Z = HHmm
range 12 years resolution 1 min legibility partial MSI yes
BinVer
Opaque. Maximum range. No pretense.
X = (s≫24) & 0xFF   Y = (s≫16) & 0xFF   Z = s & 0xFFFF
range 136 years resolution 1 sec legibility none MSI yes

How to choose

Start with SunVer unless you have a reason not to. Then:

Need Windows MSI?
 ├─ No → SunVer
 └─ Yes → Need minute precision?
    ├─ No → LightVer
    └─ Yes → Is the project lifespan < 12 years?
       ├─ Yes → MoonVer
       └─ No → BinVer

SemVer isn't wrong

SemVer communicates compatibility. Ver communicates time. They serve different purposes. Use SemVer for libraries where API contracts matter. Use Ver for applications, game builds, firmware, CI artifacts — anything where "when was this built?" is more useful than "is this backwards compatible?".
